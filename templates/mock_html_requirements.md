# Mock HTML Requirements

## Objective
Create polished, easy-to-present mock screens that show how Staffbase could look for the customer brand.

## Minimum Required Output
- one desktop mock
- one mobile mock
- brand-aligned palette
- strong headline hierarchy
- visible navigation
- card-based content blocks
- at least one CTA
- realistic sample content aligned to the customer industry

## Preferred Screen Elements
- hero / key announcement area
- news feed cards
- quick links or shortcuts
- article or detail view
- badges or highlighted state
- mobile bottom navigation
- branded menu or utility area

## Content Rules
- placeholder copy should feel realistic
- use customer-industry language where possible
- avoid lorem ipsum unless absolutely necessary
- keep content business-friendly and presentation-safe

## UI Style Rules
- clear spacing
- premium but restrained styling
- high contrast for readability
- customer-specific visual cues
- modern enterprise layout
- avoid clutter

## HTML Rules
- keep files simple to open locally
- minimize unnecessary dependencies
- use semantic HTML where practical
- separate structure from styling cleanly

## CSS Rules
- use CSS variables for palette tokens
- keep demo styling in demo-mock.css
- do not mix Staffbase production-like selectors into mock CSS
- keep classes readable and presentation-oriented

## Recommended Sections for Desktop Mock
- top header or nav
- key message area
- featured announcement
- content cards
- quick actions
- optional side panel or utility card

## Recommended Sections for Mobile Mock
- compact header
- main feed card
- quick action row
- bottom navigation
- one highlighted content item

## Review Checklist
- [ ] Looks customer-specific
- [ ] Readable from a shared screen
- [ ] Good visual contrast
- [ ] Enough realism for a sales demo
- [ ] Clean separation between mock code and Staffbase CSS proposal
